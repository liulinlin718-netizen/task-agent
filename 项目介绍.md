# TaskAgent · 项目介绍

> 面向研究生群体的 AI 任务管理桌面助手，基于 Electron + React + LLM 构建。

---

## 一、项目总览

### 1.1 一句话介绍

TaskAgent 是一款 **Local-First** 的 AI 驱动桌面任务管理工具，用户通过自然语言对话即可完成任务的增删改查、进度追踪、日报/周报生成等操作，同时提供情绪陪伴能力，定位为"科研伴侣"。

### 1.2 核心差异化

| 维度 | 传统 Todo 应用 | TaskAgent |
|:---|:---|:---|
| 交互方式 | 手动点击表单 | **自然语言对话**（"帮我加个任务"、"健完身了"） |
| 进度追踪 | 手动勾选 | **隐式识别**（"跑完步了" → 自动匹配健身任务 → 进度100%） |
| 报告生成 | 无 | **对话触发**（"总结一下这周" → 生成结构化 Markdown 报告） |
| 情绪维度 | 无 | **情绪陪伴**（识别焦虑/疲惫，回复共情而非说教） |
| 数据安全 | 云端存储 | **全本地存储 + AES-256-GCM 加密导出** |

### 1.3 技术栈

```
前端: React 19 + TypeScript + TailwindCSS 4 + Framer Motion
桌面: Electron 41（悬浮球 + 磁吸侧边栏 + 系统托盘）
AI:   OpenAI Chat Completions 兼容协议（支持 Gemini/DeepSeek/Qwen/Kimi 等）
存储: 本地 JSON 文件（fs 读写，替代 localStorage）
```

## 二、v2.0 相对于 v1.0 的核心迭代

### 2.1 统一 API 架构（消除双轨代码）

**v1.0 问题**：Gemini 用 `@google/genai` SDK + `responseSchema`，OpenAI 用 `fetch` + 手搓 JSON。每个功能写两套代码，维护成本翻倍。

**v2.0 方案**：统一为 OpenAI Chat Completions 兼容格式，只用 `fetch` 调用。利用 Gemini 的 OpenAI 兼容端点，一套代码覆盖所有供应商。

**效果**：
- 删除 `@google/genai` 依赖，`AgentService.ts` 所有 `if/else` 分支全部消除
- 新增供应商只需改 `baseUrl`，零代码改动

**评估方式**：代码审查 — 确认零 `isGemini/isOpenAI` 分支；`npx tsc --noEmit` 编译零错误。

---

### 2.2 意图识别架构重构（规则前置 + JSON 结构化输出）

**v1.0 问题**：每条消息都发给大模型做意图识别，无本地过滤。

**v2.0 方案**：两层架构 —

```
用户输入
  ↓
[第一层] matchLocalRule() — 正则/关键词匹配
  ├─ 命中 → 确定 intent，构建精简 Prompt
  └─ 未命中 ↓
[第二层] 模型兜底 — JSON 结构化输出
  └─ response_format: json_object
     → 返回 {"intent":"...", "data":{...}}
```

**6 个 Intent**：`add_tasks` / `update_task` / `delete_task` / `decompose` / `generate_report` / `chat`

**关键设计决策**：从 Function Calling（Tool 定义）改为 JSON 结构化输出。

| 维度 | Function Calling | JSON 结构化输出（最终方案） |
|:---|:---|:---|
| 固定 Token 开销 | ~600 tokens（5 个 Tool 定义） | ~200 tokens（prompt 内嵌描述） |
| 输出保证 | API 层面 `tool_calls` 字段 | API 层面 `response_format: json_object` |
| 双重输出风险 | ⚠️ 有（text + tool_call 同时出现） | ✅ 无 |

**效果**：每次请求省 ~400 tokens，消除双重输出 Bug。

**评估方式**：Token 计算对比 — v1.0 单次 2000-4000 tokens → v2.0 单次 600-1000 tokens。

---

### 2.3 滑动摘要（Rolling Summary）

**v1.0 问题**：每次请求发送完整聊天历史，10 轮对话后 Token 翻倍。

**v2.0 方案**：
- 对话超过 3 轮 → 后台异步调用小模型生成一句话摘要
- 实际发送上下文 = `摘要` + `最近 3 轮完整对话`
- 摘要缓存在 ChatSession 对象中，只在新消息超出已摘要范围时更新

```typescript
// 示例摘要输出
"用户正因为跑实验报错而沮丧，决定放弃 Python 改看文献"
```

**效果**：长对话 Token 消耗从 2500+ 降至 ~700（净节省 70%+）。

**评估方式**：对比 10 轮对话下的 prompt token 数。

---

### 2.4 上下文路由 + System Prompt 模块化

**v1.0 问题**：`systemInstruction` 是巨大模板字符串，无论闲聊还是任务操作都注入全量内容。

**v2.0 方案**：拆分为 5 个模块，按需组装：

```
mod_base_persona()       → 人设（~80 tokens）
mod_intent_instruction() → 意图识别指引 + JSON 格式（~200 tokens）
mod_task_context()       → 当日任务列表
mod_chat_instruction()   → 对话指令 + 守卫规则
mod_profile_context()    → 用户画像
```

**路由策略**：
- 规则命中 → 精简组装（persona + intent + tasks + chatContext）
- 模型兜底 → 完整组装（+ chat_instruction + profile）

**关键设计**：`task_context` 始终注入 + chat 守卫指令。原因：不注入时模型无法识别"健完身了"这类隐式任务进展。守卫指令防止模型主动提及任务列表。

---

### 2.5 Per-Agent 模型配置

**v1.0 问题**：单一 API Key + 单一模型，闲聊和深度报告用同一个模型。

**v2.0 方案**：
- **Chat Agent**：快速模型（如 `gemini-2.0-flash`），低延迟
- **Report Agent**：深度模型（如 `gemini-2.5-flash`），高质量，60s 超时
- 报告 API 留空则自动沿用全局配置

---

### 2.6 桌面集成（悬浮球 + 磁吸 + 托盘）

| 功能 | v1.0 | v2.0 |
|:---|:---|:---|
| 拖拽 | 偶有卡顿 | `requestAnimationFrame` 节流，稳定 60fps |
| 磁吸 | 无反馈 | 靠近边缘 → 球体 inset 内发光提示 |
| 展开/收起 | 瞬间跳变 | `easeOutCubic` 200ms 缓动 |
| 关闭行为 | 直接退出 | 最小化到系统托盘，后台常驻 |
| 进度控制 | 整行滚轮 | 仅进度环区域（`passive:false` + `preventDefault`） |

---

### 2.7 其他功能

- **文档解析**：支持 .txt/.docx/.pdf → 提取待办事项 → 批量确认添加
- **数据存储**：localStorage → Electron fs JSON（无容量上限）
- **加密导出**：AES-256-GCM + PBKDF2 10万次迭代
- **报告同步**：对话中生成的报告自动同步到历史报告列表
- **报告排序**：按目标日期降序 + 时间滚轮连续日期填充

### 4.2 System Prompt 动态组建

```typescript
function buildSystemPrompt(state, ruleHint) {
  // 公共模块
  const persona    = mod_base_persona(state);      // "You are 任务助理..."
  const taskCtx    = mod_task_context(state);       // "[Today's Tasks]\nabc: 健身 (0%)"
  const chatCtx    = buildChatContext(session);     // 摘要 + 最近3轮

  if (ruleHint) {
    // 规则命中 → 精简 Prompt（省略 chat_instruction 和 profile）
    return [persona, mod_intent_instruction(), taskCtx, chatCtx].join("\n\n");

  // 模型兜底 → 完整 Prompt
  return [persona, mod_intent_instruction(), mod_chat_instruction(),

}
```

## 四·B、Agent 架构详解（Function Calling 版）

> 以下是 v2.0 中间版本采用的 Function Calling 架构，后因 Token 开销和双重输出问题回退为 JSON 结构化输出。保留此节以展示设计决策过程。

### 4B.1 核心思路

Function Calling（FC）是 OpenAI 提出的标准化工具调用协议：开发者在请求中定义 `tools` 数组，模型决定是否调用某个 Tool，若调用则返回结构化的 `tool_calls` 而非自由文本。

**在 TaskAgent 中的应用**：将 5 个任务操作定义为 Tool，模型不调用任何 Tool 时即为闲聊（`chat` 是天然兜底）。

### 4B.2 请求格式

```json
{
  "model": "gemini-2.5-flash",
  "stream": true,
  "messages": [
    { "role": "system", "content": "You are 任务助理..." },
    { "role": "user",   "content": "健完身了" }
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "add_tasks",
        "description": "当用户想要添加新任务时调用",
        "parameters": {
          "type": "object",
          "properties": {
            "proposedTasks": { "type": "array", "items": { "type": "string" } },
            "targetDate":    { "type": "string" }
          },
          "required": ["proposedTasks"]
        }
      }
    },
    {
      "type": "function",
      "function": {
        "name": "update_task",
        "description": "更新任务进度、日期、备注或优先级",
        "parameters": {
          "type": "object",
          "properties": {
            "taskId":   { "type": "string" },
            "progress": { "type": "number" },
            "date":     { "type": "string" },
            "notes":    { "type": "string" },
            "priority": { "type": "string", "enum": ["low","medium","high"] }
          },
          "required": ["taskId"]
        }
      }
    },
    { "...delete_task..." },
    { "...decompose..." },
    { "...generate_report..." }
  ]
}
```

**每个 Tool 定义约 100-150 tokens，5 个合计 ~600 tokens，每次请求固定携带。**

### 4B.3 响应处理（SSE 流）

模型返回 SSE（Server-Sent Events）流，客户端需要区分三种情况：

**情况 A — 模型调用 Tool**（任务操作）：
```
data: {"choices":[{"delta":{"tool_calls":[{"function":{"name":"update_task"}}]}}]}
data: {"choices":[{"delta":{"tool_calls":[{"function":{"arguments":"{\"taskId\":\"abc\""}}]}}]}
data: {"choices":[{"delta":{"tool_calls":[{"function":{"arguments":",\"progress\":100}"}}]}}]}
data: [DONE]
```
客户端逐 chunk 拼接 `arguments` → `[DONE]` 后 `JSON.parse` → 根据 `function.name` 分发执行。

**情况 B — 模型不调用 Tool**（闲聊）：
```
data: {"choices":[{"delta":{"content":"你"}}]}
data: {"choices":[{"delta":{"content":"好"}}]}
data: {"choices":[{"delta":{"content":"！"}}]}
data: [DONE]
```
客户端逐 chunk 追加文本 → 打字机效果。

**情况 C — 双重输出（Bug）**：
```
data: {"choices":[{"delta":{"content":"好的，我来帮你"}}]}   ← 先产生文字
data: {"choices":[{"delta":{"tool_calls":[...]}}]}            ← 又开始调 Tool
data: [DONE]
```
模型同时产生了文字和 tool_call → 用户先看到文字再被丢弃 → 重新执行 Tool 生成新回复 → **双重输出，浪费 token，体验差**。

### 4B.4 FC 版完整流程图

```
用户输入
  ↓
matchLocalRule() — 正则匹配
  ├─ 命中 → 确定 ruleHint
  └─ 未命中 → ruleHint = null
  ↓
buildSystemPrompt(state, ruleHint)
  ↓
callChatCompletion({
  stream: true,
  tools: [add_tasks, update_task, delete_task, decompose, generate_report]
})
  ↓
SSE 流解析 (StreamParser)
  ├─ 收到 content delta → 逐字渲染到 UI（打字机）
  ├─ 收到 tool_calls delta → 拼接 arguments JSON
  └─ 收到两者 → ⚠️ 丢弃已渲染文字，只保留 tool_call
  ↓ [DONE]
JSON.parse(拼接后的 arguments)
  ↓
分发执行: function.name → addTask / updateTask / deleteTask / ...
  ↓
生成回复消息展示给用户
```

### 4B.5 StreamParser 实现要点

```typescript
// StreamParser.ts（FC 版核心逻辑，v2.0.1 已弃用）
export async function parseSSEStream(
  response: Response,
  onTextChunk: (text: string) => void,
  onToolCall: (name: string, args: any) => void,
) {
  const reader = response.body!.getReader();
  const decoder = new TextDecoder();
  let toolName = '';
  let toolArgs = '';
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });

    // 按行解析 SSE
    const lines = buffer.split('\n');
    buffer = lines.pop() || '';

    for (const line of lines) {
      if (!line.startsWith('data: ')) continue;
      const data = line.slice(6);
      if (data === '[DONE]') break;

      const parsed = JSON.parse(data);
      const delta = parsed.choices?.[0]?.delta;

      // 文字内容
      if (delta?.content) onTextChunk(delta.content);

      // Tool call 增量拼接
      if (delta?.tool_calls?.[0]) {
        const tc = delta.tool_calls[0];
        if (tc.function?.name) toolName = tc.function.name;
        if (tc.function?.arguments) toolArgs += tc.function.arguments;
      }
    }
  }

  if (toolName) {
    onToolCall(toolName, JSON.parse(toolArgs));
  }
}
```

### 4B.6 FC 版 vs JSON 版对比总结

| 维度 | Function Calling (FC) | JSON 结构化输出（当前） |
|:---|:---|:---|
| **固定 Token** | ~600 tokens（5 个 Tool Schema） | ~200 tokens（prompt 内嵌） |
| **输出格式保证** | API 层面 `tool_calls` 字段 | API 层面 `response_format: json_object` |
| **流式首字** | ✅ < 1s（原生 SSE） | ❌ ~1-2s（非流式+客户端打字机） |
| **双重输出** | ⚠️ 有（需额外处理） | ✅ 不可能发生 |
| **解析复杂度** | 高（SSE 逐 chunk 拼接 + 分支） | 低（一次 `JSON.parse`） |
| **代码量** | 多（StreamParser ~160 行） | 少（parseJsonResponse ~20 行） |
| **新增 intent 成本** | +100 tokens/个 Tool | +30 tokens/个 prompt 描述 |
| **适用场景** | Tool 数量多(10+)、参数复杂、需要多步调用 | Tool 数量少(≤6)、参数简单、单步调用 |

## 五、评测与效果

### 5.1 Token 效率

| 场景 | v1.0 | v2.0（最终） | 节省比例 |
|:---|:---|:---|:---|
| 固定开销（意图指引） | ~200 tokens | ~200 tokens | 0%（与 v1.0 持平） |
| 短对话（1-4 轮） | 500-800 | 600-800 | ~0%（持平） |
| 长对话（10+ 轮） | 2500+ | ~700 | **~72%** |
| 单次请求总计 | 2000-4000 | 600-1000 | **~65%** |

> 中间版本（Function Calling）固定开销为 ~600 tokens，反而比 v1.0 更贵，因此回退为 JSON 结构化输出。

### 5.1.1 JSON 结构化输出的缺点（相对于 Function Calling）

| 缺点 | 说明 |
|:---|:---|
| **无原生流式首字** | FC 支持 `stream: true`，文字内容逐字返回，TTFT < 1s。JSON 模式必须等完整 JSON 生成完毕才能解析，TTFT ~1-2s，需要客户端模拟打字机弥补。 |
| **无 Schema 强约束** | FC 的 Tool 定义通过 JSON Schema 精确约束每个参数的类型（string/number/enum），模型不符合 Schema 会被 API 拒绝。JSON 模式只保证"输出是合法 JSON"，不保证字段名和类型正确，需要客户端做 fallback 处理。 |
| **扩展性有限** | 当 intent 数量超过 10 个、参数结构嵌套复杂时，prompt 内的自然语言描述会变得冗长且模糊，不如 JSON Schema 精确。FC 天然适合大规模工具编排场景。 |
| **无多步调用能力** | FC 支持模型在一次请求中返回多个 `tool_calls`（并行调用），也支持将 Tool 执行结果回传给模型做下一步推理（Agentic loop）。JSON 模式只能返回一个 intent，无法做多步推理。 |
| **依赖 prompt 工程质量** | FC 的参数约束由 Schema 保证，与 prompt 措辞无关。JSON 模式的 intent 识别准确率高度依赖 `mod_intent_instruction()` 的自然语言描述质量，措辞不当可能导致误判。 |

**结论**：JSON 结构化输出适合本项目（intent ≤ 6 个、参数简单、单步调用），但不适合需要大量工具、复杂参数、多步推理的通用 Agent 框架。

### 5.2 响应速度

| 指标 | v1.0 | v2.0 |
|:---|:---|:---|
| TTFT（首字延迟） | 2-5s（等全量返回） | ~1-2s（非流式+客户端打字机） |
| 报告生成 | 经常超时失败 | 60s 超时，稳定生成 |

## 六、面试讲述建议

### 30 秒版本

> "我做了一个 AI 驱动的桌面任务管理工具，基于 Electron + React + LLM。用户通过自然语言对话管理任务，比如说'健完身了'就能自动匹配健身任务并更新进度。核心技术点是设计了一套两层意图识别架构（正则前置 + JSON 结构化输出），配合滑动摘要将长对话 Token 消耗降低了 70%。"

### 2 分钟版本

在 30 秒版本基础上，补充：

1. **为什么不用 Function Calling**："我一开始用了 OpenAI 的 Function Calling，但发现 5 个 Tool 定义每次请求固定多花 600 tokens，而且 stream+tools 会出现双重输出 Bug。改成 JSON 结构化输出后，用 `response_format: json_object` 从 API 层面保证格式，prompt 内用自然语言描述 intent，省了 400 tokens/次。"

2. **滑动摘要怎么做的**："对话超过 3 轮后，后台异步调小模型把旧消息压缩成一句话摘要，缓存在 session 里。实际发给模型的是'摘要 + 最近 3 轮'，长对话从 2500 tokens 降到 700。"

3. **上下文路由的权衡**："一开始设计是闲聊不注入任务列表以省 Token，但发现用户说'健完身了'时模型不知道有健身任务，无法匹配。最终改为始终注入 + chat 守卫指令（'不要主动提及任务列表'），兼顾了识别准确率和自然感。"
