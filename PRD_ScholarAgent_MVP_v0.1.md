# ScholarAgent · 最小可行产品 (MVP) 验证文档

> **文档性质**：本文档为 ScholarAgent 立项初期的 MVP 验证方案，目标是用最低成本、最短周期验证核心产品假设。本文档不追求功能完备，只追求 **"能不能跑通核心闭环"**。

---

## 一、背景与动机

### 1.1 一句话描述

> 做一个桌面端的 AI 任务助手，让研究生用"说话"的方式管理任务，而不是手动填表单。

### 1.2 为什么要做这个？

在日常学习和科研中，我观察到一个普遍现象：

- 研究生的任务不是"买菜""取快递"这种一次性的事，而是"写论文第三章""调模型参数"这种 **持续推进** 的事。
- 现有的 Todo 工具（滴答清单、Todoist）只有"完成/未完成"两个状态，根本没法表达"我今天推进了 30%"。
- 每次记录任务都要 **切窗口、打开应用**，严重打断写论文/写代码的心流。

### 1.3 核心假设（需要验证）

| 编号 | 假设 | 验证标准 |
| :--- | :--- | :--- |
| H1 | 用户愿意用"自然语言"代替"手动表单"来添加任务 | 用户能通过对话成功添加至少 3 条任务 |
| H2 | "进度滑块"（0-100%）比"打钩"更适合学术任务 | 用户主动使用滑块调整进度，而非只在 0% 和 100% 之间切换 |
| H3 | AI 能够准确理解用户意图并执行对应操作 | 意图识别准确率 > 80%（在 add/update/chat 三种场景下） |

## 二、MVP 范围定义

### 2.1 只做什么（In Scope）

**一个能用的网页端（先不做桌面端），包含以下最小功能集：**

1. **任务列表**：显示当天的任务，每个任务有名称和一个 0-100% 的进度滑块
2. **AI 对话框**：一个输入框，用户打字后 AI 回复，并能自动添加任务或更新进度
3. **意图识别**：AI 能区分三种意图：
   - `add_tasks`：用户想添加新任务
   - `update_progress`：用户想更新某个任务的进度
   - `chat`：用户只是闲聊
4. **本地存储**：数据存在浏览器 localStorage 里，刷新不丢失

### 2.2 明确不做什么（Out of Scope）

以下功能 **本轮不做**，留给验证通过后的迭代：

- ❌ 桌面端 Electron 打包（先用浏览器跑）
- ❌ 悬浮球、侧边栏等花哨交互
- ❌ 多日历视图、历史回溯
- ❌ 用户档案、头像上传
- ❌ 多模型切换（先只用 Gemini）
- ❌ 日结转 / Rollover 功能
- ❌ 暗色模式
- ❌ 精美 UI（能用就行）

### 2.3 技术选型（最小化）

| 层面 | 选择 | 理由 |
| :--- | :--- | :--- |
| 前端框架 | React + Vite | 最快的开发启动速度 |
| 样式 | Tailwind CSS | 不需要写 CSS 文件，直接在 HTML 里写样式 |
| AI 模型 | Gemini Flash | 免费额度高、响应快、支持 JSON 结构化输出 |
| 数据存储 | localStorage | 零成本，无需搭建后端 |
| 部署 | 本地 dev server | MVP 阶段不需要部署到线上 |

## 三、核心交互流程

### 3.1 唯一的用户旅程

```
用户打开网页
  │
  ├── 左侧：看到今天的任务列表（初始为空）
  │     └── 每个任务 = 名称 + 进度滑块(0-100%)
  │
  └── 右侧：AI 对话框
        │
        ├── 用户输入："帮我添加一个任务：写开题报告"
        │     └── AI 识别 intent=add_tasks → 任务列表自动新增一条
        │
        ├── 用户输入："开题报告写了一半了"
        │     └── AI 识别 intent=update_progress → 自动把进度滑到 50%
        │
        └── 用户输入："好累啊不想干了"
              └── AI 识别 intent=chat → 返回鼓励性回复
```

## 四、AI 意图识别方案（最简版）

### 4.1 Prompt 设计

给 Gemini 的 System Prompt（核心只有这一段）：

```
你是一个任务管理助手。根据用户的输入，判断他的意图，并返回 JSON 格式：

{
  "intent": "add_tasks" | "update_progress" | "chat",
  "reply": "你的回复内容",
  "taskName": "任务名称（仅 add_tasks 时）",
  "taskId": "任务ID（仅 update_progress 时）",
  "progress": 50（仅 update_progress 时，0-100 的数字）
}

当前用户的任务列表：
{{TASKS_JSON}}
```

### 4.2 为什么用 JSON 而不是自由文本？

- 自由文本需要正则匹配，非常不稳定
- Gemini 支持 `responseSchema` 可以强制输出 JSON 格式
- JSON 可以直接 `JSON.parse()` 后写入 State，零解析成本

## 五、验收标准

### 5.1 MVP 算"验证通过"的条件

- [ ] 能通过对话成功添加 3 条以上任务
- [ ] 能通过对话成功更新任务进度（如"完成了一半" → 50%）
- [ ] 闲聊时不会误触发任务操作
- [ ] 刷新页面后数据不丢失
- [ ] 整体交互流程顺畅，无明显卡顿

### 5.2 验证通过后的下一步

如果以上假设全部验证通过，立即启动以下迭代：

1. **优先级 P0**：接入 Electron，实现桌面端 + 悬浮球交互
2. **优先级 P1**：增加任务拆解能力（`decompose` 意图）
3. **优先级 P1**：增加日历视图，支持多日期切换
4. **优先级 P2**：增加日结转（Rollover）和生物钟设置
5. **优先级 P2**：增加多模型兼容（OpenAI 协议）

## 七、风险与应对

| 风险 | 影响 | 应对 |
| :--- | :--- | :--- |
| Gemini API 意图识别不准 | 添加任务失败或误操作 | 使用 `responseSchema` 强制 JSON 输出；识别失败时降级为 chat |
| localStorage 容量有限 | 长期使用数据丢失 | MVP 阶段可接受；后续迁移到 IndexedDB 或文件系统 |
| 用户说的话太模糊 | AI 无法判断意图 | 在回复中引导用户明确表达（"你是想添加任务还是更新进度？"） |
