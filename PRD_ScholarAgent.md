## 三、名词解释

| 术语 | 定义 |
| :--- | :--- |
| **Orb（悬浮球）** | 桌面端常驻的透明无边框圆形窗口，作为 AI 助手的物理化身，支持拖拽、磁吸、点击展开对话面板。 |
| **Task Drawer（任务抽屉）** | 桌面边缘的磁吸式任务侧边栏窗口，吸附时仅显示 6px 的色带指示条，悬停后自动展开为完整面板。 |
| **柔性重置（Rollover）** | 打破传统凌晨 0 点重置任务的硬性逻辑，允许用户自定义"新一天"的起点（如凌晨 2:00），未完成任务自动结转至次日并保留已有进度。 |
| **意图路由（Intent Routing）** | Chat Agent 对用户自然语言输入进行分析，输出结构化的 `intent` 字段（add_tasks / update_progress / decompose / chat），驱动不同的 UI 响应流。 |
| **Local-First** | 本地优先架构。用户产生的全量数据（任务、对话、个人档案、报告）仅存储于本地浏览器 localStorage，不经过任何开发者服务器，保障科研数据绝对隐私。 |
| **IPC（进程间通信）** | Electron 架构下主进程（Node.js）与渲染进程（浏览器）之间的消息传递机制，通过 contextBridge 实现安全沙箱隔离。 |

## 四、需求背景

### 4.1 市场现状与产品定位

当前效率工具市场正处于从 **"记录工具（System of Record）"** 向 **"执行系统（System of Agency）"** 跨越的拐点。传统 Todo 应用停留在被动记录层面，无法提供主动的认知辅助。ScholarAgent 瞄准这一转型窗口，定位为 **AI-Native 的桌面级任务伴侣**——不仅帮用户"记住"，更帮用户"想清楚"和"做下去"。

### 4.2 目标用户画像

| 维度 | 描述 |
| :--- | :--- |
| **核心用户** | 在读硕博研究生、高校科研人员、独立研究者 |
| **扩展用户** | 知识型自由职业者、产品经理、内容创作者 |
| **典型场景** | 论文撰写、文献调研、实验迭代、课题拆解、日常事务管理 |
| **设备环境** | 长时间使用台式/笔记本电脑，需要桌面常驻且不打断心流的轻量工具 |

### 4.3 用户调研：核心痛点分析

**痛点一：任务的非二元属性**
> 研究生的核心任务（写论文、跑模型、调实验）是长周期、高模糊度的过程型工作。传统 Todo 工具的"打钩 / 未打钩"（即 0% 或 100%）机制无法表达"我今天推进了 30%"的真实状态，带来强烈的挫败感与虚无感。

**痛点二：生物钟错位引发的节点焦虑**
> 学术人群普遍存在晚睡习惯（凌晨 1-3 点），传统工具在 0:00 强行刷新任务列表，导致用户在深夜高效工作后看到空白列表，产生"今天一事无成"的错觉焦虑。

**痛点三：执行阻力与心流中断**
> 记录任务需要切换窗口、打开浏览器、登录账号，严重打断研究心流。面对庞大课题时无从下手，缺乏系统化的拆解辅助和情绪疏导。

**痛点四：隐私顾虑**
> 科研数据（实验思路、论文进展）具有高度敏感性，用户对云端同步存在信任焦虑。

### 4.4 竞品分析

| 竞品类型 | 代表产品 | 优势 | 劣势 |
| :--- | :--- | :--- | :--- |
| **传统派** | Todoist, Things 3, 滴答清单 | 逻辑清晰，跨平台 | 无主动智能，无法拆解任务，缺乏情绪伴随，强制二元状态 |
| **All-in-one 派** | Notion, 飞书多维表格 | 功能全面 | 过于沉重，配置成本极高，非桌面原生，启动慢、打断心流 |
| **AI 对话派** | ChatGPT, Kimi | 智能对话强 | 无持久化任务状态，对话即丢，无桌面集成 |

**ScholarAgent 核心差异化破局点：**
1. **桌面极简原生**：悬浮球 + 磁吸侧边栏，随叫随到、用完即走
2. **进度滑块替代打钩**：量化模糊任务，正向反馈驱动
3. **意图驱动的双 Agent 协同**：自然语言直接操控任务状态
4. **Local-First 隐私架构**：零云端传输，数据绝对本地

| 闭环 | 说明 | 状态 |

## 六、功能详细说明

### 6.1 产品流程图：核心用户旅程

1. **启动**：用户打开应用，主窗口三栏布局加载（左导航 + 中部内容 + 右侧AI对话）
2. **悬浮球唤醒**（可选）：桌面任意位置拖拽，松手自动磁吸屏幕边缘；点击展开380x520暗色毛玻璃对话面板；失焦400ms后自动收起（可固定Pin）
3. **任务抽屉**（可选）：拖拽至屏幕边缘自动吸附为6px色带指示条；鼠标悬停色带展开完整任务面板；滚轮微调进度（每滚一格 ±1%）
4. **意图识别**：用户自然语言输入，Agent识别意图后自动操控任务
5. **夜间结转**：到达预设生物钟节点，RolloverEngine 自动克隆未完成任务并生成AI摘要

### 6.2 交互原型：窗口架构设计

**设计语言**：Apple 级极简暗色毛玻璃风格，backdrop-filter: blur(24px) + 半透明渐变背景。

#### 6.2.1 主窗口（mainWindow）
- **尺寸**：默认 1280x800，最小 900x600
- **左导航**（80px）：图标式导航，带 layoutId 驱动的弹簧动画活跃指示器
- **中部内容区**（弹性宽度）：承载四个页面（任务中心 / 历史与总结 / 个人档案 / 设置）
- **右侧对话栏**（默认 380px）：可拖拽分隔条调节宽度（250-600px），实时 AI 对话
- **主题**：支持浅色 / 深色模式全局切换

#### 6.2.2 悬浮球窗口（ballWindow）
- **收起态**：48x48 透明无边框窗口，蓝色渐变圆形 + 呼吸光环动画
- **展开态**：380x520 暗色毛玻璃面板，内嵌完整 AI 对话能力
- 拖拽使用 requestAnimationFrame 优化，区分"点击"与"拖拽"（位移 > 3px 判定为拖拽）
- 松手后基于中心点判定磁吸方向（距边缘 < 100px 触发），0 像素贴合
- Pin 功能可固定面板，阻止失焦收起

#### 6.2.3 任务抽屉窗口（taskCenterWindow）
- **默认态**：320x480，可自由调整尺寸（260-500 宽，300-800 高）
- **吸附态**：6px 宽的蓝色渐变色带，覆盖屏幕边缘全高
- **展开态**：鼠标悬停色带后，从边缘弹出完整面板
- **数据同步**：通过 StorageEvent 监听 localStorage 变化，实现跨窗口实时数据同步

### 6.3 功能清单

#### F1. 柔性进度任务体系（TaskHub）
- **进度滑块**：替代传统 Checkbox，0-100% 连续滑动，实时反馈
- **双击重命名**：双击任务名称进入编辑模式，失焦或回车自动保存
- **优先级标签**：三级循环切换：低（绿）-> 中（橙）-> 高（红）
- **任务备注**：可展开/收起的 Notes 区域，支持 Markdown 记录
- **日历导航**：5 日横向滚轴（弹簧动画 + 缩放效果）+ Popover 日历快速跳转
- **任务删除**：Hover 显示删除按钮，防误触设计

#### F2. 生物钟自定义（Chronotype Settings）
- 用户在设置页配置"日结算点"（默认凌晨 02:00），精确到分钟
- 支持滚轮微调小时和分钟值
- RolloverEngine 每 60 秒轮询一次，判断当前"逻辑日期"
- 触发结转时：自动克隆未完成任务（保留已有进度）至新一天，并调用 Report Agent 生成昨日 AI 摘要

#### F3. 全局历史溯源（History）
- **多维度视图**：日 / 周 / 月 / 年四种聚合粒度切换
- **任务快照**：每个历史日期展示当日全部任务及其最终进度
- **AI 日摘要**：Rollover 自动生成的简短激励性摘要
- **自定义报告生成器**：浮动操作栏，用户选择任意日期范围，AI 生成结构化 Markdown 分析报告
- **报告管理**：报告列表（卡片式网格布局）、详情查看、滚动导航条

#### F4. 个人档案系统（Profile）
- **头像上传**：支持图片裁剪（集成 react-easy-crop），圆角矩形裁剪框
- **基本信息**：职业与领域、核心技能栈
- **详细描述**：个人简介（Bio）、当前核心目标（Goal）
- **AI 联动**：Profile 数据在满足条件时动态注入 System Prompt，但附带严格指令避免"读心术"体验

#### F5. AI 对话系统（AgentChat）
- **多会话管理**：支持新建、切换、删除对话会话，侧滑动画历史面板
- **自动标题**：首次对话后由 AI 自动生成 3-10 字的会话标题
- **任务卡片推荐**：AI 返回 proposedTasks 后，在对话中内嵌可操作的任务卡片
- **单条 / 全部应用**：用户可逐条或一键全部将推荐任务添加到任务中心
- **刷新推荐**：对已有推荐任务点击"刷新"，AI 重新生成完全不同的建议
- **重新生成**：对最后一条 AI 回复进行重新生成（自动排除旧回复避免上下文污染）
- **停止生成**：AbortController 实现的即时中断能力
- **一键切换人格**：对话输入框下方色条，点击循环切换：专业导师（蓝）-> 贴心助手（绿）-> 严厉督导（橙）
- **Markdown 渲染**：基于 react-markdown 的富文本渲染

#### F6. 系统设置（Settings）
- **外观**：浅色 / 深色主题切换
- **生物钟**：日结算点配置
- **助理名称**：自定义 AI 名字（全局生效）
- **助理模式**：专业导师 / 贴心助手 / 严厉督导
- **API 配置**：支持 Google Gemini / OpenAI / DeepSeek / 千问 Qwen / Kimi Moonshot / 智谱 GLM / MiniMax / 豆包 Doubao 共 8 家服务商；自定义 Base URL（支持本地 Ollama 等私有化部署）
- **数据管理**：清理缓存（仅清对话和报告）/ 清理全部数据（不可逆）


## 七、非功能需求

### 7.1 性能需求

| 指标 | 目标值 | 实现方式 |
| :--- | :--- | :--- |
| AI 首字响应（TTFT） | < 1500ms | 直接调用 Gemini / OpenAI API，无中间代理层 |
| 悬浮球拖拽帧率 | 60fps | requestAnimationFrame 节流 + Electron setBounds 直接操作 |
| 跨窗口数据同步延迟 | < 100ms | StorageEvent 原生监听，无轮询开销 |
| 冷启动时间 | < 3s | Vite 构建优化 + 预加载脚本并行初始化 |
| 安装包体积 | ~115MB | NSIS 压缩，安装后约 212MB（含 Chromium 运行时） |

### 7.2 隐私与安全需求（Privacy by Design）

| 设计原则 | 实现方式 |
| :--- | :--- |
| **Local-First 数据存储** | 全量数据通过 localStorage（key: scholaragent-state）持久化于本地 AppData 目录，不经过任何开发者服务器 |
| **API Key 客户端隔离** | Key 仅保存在用户本地，通过设置页手动输入，源码与 Git 仓库中无任何硬编码凭证 |
| **沙箱安全** | Electron contextIsolation: true + nodeIntegration: false，渲染进程无法直接访问 Node.js API |
| **安全桥接** | 所有系统级操作通过 contextBridge.exposeInMainWorld 暴露最小权限 API |
| **错误隔离** | 渲染进程全局错误（window.onerror + console.error）转发至主进程日志，不暴露给最终用户 |

### 7.3 兼容性与可扩展性

- **多平台**：当前发布 Windows NSIS 安装包，Electron 架构天然支持 macOS / Linux 扩展
- **多模型**：代码层抽象了 Gemini 原生协议与 OpenAI 兼容协议两条调用链路，新增服务商仅需添加一个 SelectItem
- **多语言**：当前 UI 为中文，Prompt 层已具备 i18n 扩展基础

## 八、模型与 Agent 架构（核心竞争力）

### 8.1 模型策略

| 维度 | 策略 |
| :--- | :--- |
| **默认引擎** | Gemini 2.5 Flash（高速、低成本、原生 JSON Schema 支持） |
| **兼容引擎** | OpenAI 协议兼容层，支持 GPT-4o-mini / DeepSeek / Qwen / Kimi / GLM / MiniMax / Doubao 等 8 家服务商 |
| **私有化部署** | 支持自定义 Base URL，可接入 Ollama、vLLM 等本地推理引擎 |
| **选型原则** | 优先低延迟、高并发模型，保障桌面级实时交互体验 |

### 8.2 双 Agent 架构（职责分离模式）

本系统采用 **职责分离的双 Agent 架构**，通过两个独立的 LLM 调用函数实现不同场景的智能处理：

#### 8.2.1 交互 Agent（Chat Agent · processAgentRequest）

**职责定位**：前台实时交互智能体，负责意图识别、任务操控与情绪陪伴。

**意图路由表**：

| Intent | 触发条件 | Agent 行为 | UI 响应 |
| :--- | :--- | :--- | :--- |
| add_tasks | 用户要求添加任务 | 提取任务实体，自动生成 2 个关联推荐。支持跨日期添加 | 首条任务自动加入任务中心，其余以可操作卡片展示 |
| update_progress | 用户描述进展 | 语义匹配最佳 taskId，设置进度值 0-100 | 自动滑动对应任务卡片的进度条 |
| decompose | 用户要求拆解任务 | 将大任务分解为可操作的子步骤 | 以可操作任务卡片列表展示，支持逐条或批量添加 |
| chat | 日常闲聊 / 情绪疏导 | 生成自然语言回复 | Markdown 富文本渲染 |

**输出约束**（反幻觉机制）：
- **Gemini 引擎**：使用原生 responseSchema（Type.OBJECT）强制 JSON 结构化输出，在 API 层面杜绝格式偏差
- **OpenAI 兼容引擎**：在 System Prompt 末尾追加严格的 JSON 格式指令，辅以 cleanJSON 正则清洗，双重保障解析成功率

**自动标题生成**：当会话为新建且消息不超过 2 条时，要求 Agent 在 chatTitle 字段返回 3-10 字的会话摘要。

#### 8.2.2 分析 Agent（Report Agent · generateCustomSummary）

**职责定位**：后台分析智能体，在两种场景下被唤起：

| 场景 | 触发方式 | 输出 |
| :--- | :--- | :--- |
| **每日自动结转** | RolloverEngine 检测到逻辑日期跳变 | 2 句话的简短激励性日摘要 |
| **用户主动请求** | 在"历史与总结"页面选择日期范围后点击"生成总结" | 完整的结构化 Markdown 分析报告 |

**报告生成规范**（基于 Prompt 约束的 5 段式结构）：
1. 整体概览
2. 任务进度审计（总结性概述，非逐条罗列）
3. 关键问题和建议
4. 抓紧行动（敦促用户的待办事项）
5. 结语

**报告生成原则**（写入 System Prompt）：
- 拒绝爹味与说教：充分理解多线并行的压力
- 包容多样性：任务可能涉及生活、求职、娱乐，不强行与学术挂钩

### 8.3 提示词工程详细设计（Prompt Engineering）

#### 8.3.1 人格系统

| 人格模式 | 标识 | 行为描述 | UI 对应色 |
| :--- | :--- | :--- | :--- |
| 专业导师 | academic | 严谨、结构化、学术风格回复 | 蓝色 |
| 贴心助手 | gentle | 温暖、共情、鼓励式回复 | 绿色 |
| 严厉督导 | strict | 直接、高标准、督促式回复 | 橙色 |

人格标识动态注入 System Prompt 的 agentStyle 字段，影响 Agent 的回复风格。用户可在设置页全局配置，也可在对话输入框下方的色条一键循环切换。

#### 8.3.2 上下文架构（分层记忆系统）

| 层级 | 内容 | 注入时机 | 目的 |
| :--- | :--- | :--- | :--- |
| **Session 上下文** | 当前会话的全部历史消息（user + model 交替） | 每次请求 | 保持对话连贯性 |
| **任务上下文** | 当前活跃日期的全部任务列表（id, name, progress, notes） | 每次请求 | 使 Agent 能精准匹配 taskId |
| **用户画像上下文** | Profile 中的 major, goal, skills, bio | 仅当字段非空时注入，附带"非主动提及不得使用"指令 | 个性化但不侵入 |
| **时间上下文** | 当前活跃日期 | 每次请求 | 支持日期感知的任务操作 |

#### 8.3.3 任务名称约束

所有由 Agent 生成的 proposedTasks 被约束为 **不超过 20 个字符**，确保在任务卡片 UI 中不溢出。

## 附录 A：面试常见问题预案

**Q1：为什么选择 Electron 而不是 Web 应用？**
> 产品核心差异化在于"桌面级伴随感"。悬浮球常驻、磁吸交互、多窗口联动、边缘触发等能力依赖系统级 API（BrowserWindow、screen、alwaysOnTop），这是纯 Web 应用无法实现的。Electron 牺牲了一定的包体积，但换来了不可替代的交互范式。

**Q2：如何防止大模型幻觉导致任务数据错乱？**
> 采用了两层防御机制：(1) Gemini 引擎使用原生 responseSchema 在 API 层面强制 JSON 结构化输出，字段类型和枚举值均有严格约束；(2) OpenAI 兼容引擎在 Prompt 末尾追加格式指令，并在客户端进行 JSON.parse + 异常兜底（解析失败时降级为普通 chat 回复），确保任何情况下 UI 都不会崩溃。

**Q3：Local-First 存储会有容量瓶颈吗？**
> localStorage 的上限约 5-10MB，对于纯文本任务记录来说可支撑数年使用。Phase 2 规划中已纳入迁移至 IndexedDB 或 electron-store 的方案，以支持大数据量场景。

**Q4：悬浮球和侧边栏的窗口控制是如何解耦的？**
> 通过独立的 IPC 通道实现：app:update-ball 和 app:update-taskcenter 各自维护独立的 prevVisible 状态标志。仅在"从关到开"的状态转换时重置窗口位置，"从开到开"时保持用户自定义位置不变，彻底避免了互相干扰。

**Q5：这个项目体现了你的哪些能力？**
> (1) **产品定义能力**：从用户痛点出发，设计了"进度滑块替代打钩"和"生物钟柔性重置"两个差异化创新点；(2) **AI 工程能力**：设计了意图路由驱动的双 Agent 架构，并实现了反幻觉的结构化输出约束；(3) **全栈工程能力**：独立完成了从 React 前端到 Electron 桌面端到 NSIS 安装包打包的完整工程闭环；(4) **交互设计能力**：实现了磁吸算法、多窗口联动、毛玻璃 UI 等桌面级精致交互。
